# 现况

随着项目业务的增长，组件越来越多，webpack 编译速度也是越来越慢，第一次启动编译速度达到恐怖的 100s ，而第一次打包时间达到了 120s ，所以有了这一次的升级优化之旅。

# 分析

由于目前webpack的版本还是几年前的 webpack 3 版本，webpack 3 的代码压缩，代码编译速度，插件支持度，模块解析速度都落后于webpack4 ，所以首当其冲先升级 webpack4 ，其次再找出哪些步骤耗时比较多。

# mode

webpack4 中新增 mode 选项 , 可选参数有 development | production | none , 其默认值为 production , mode 选项的值会将 DefinePlugin 中 process.env.NODE_ENV 的值同步，所以不需要再手动设置 DefinePlugin 。 而我司项目中 process.env.NODE_ENV 除了 development 和 production 还有使用到 test ，但是 mode 选项并不支持 test 这里使用 cross-env 重新定义一个 ENV 为 BUILD_ENV 。

- "dev": "node server/server.js",
- "start": "cross-env NODE_ENV=production node server/server.js",
- "start-pre": "cross-env NODE_ENV=production cross-env PRE_ENV=true node server/server.js",
- "test": "cross-env NODE_ENV=test node server/server.js",
+ "dev": "cross-env NODE_ENV=development BUILD_ENV=dev node server/server.js",
+ "start": "cross-env NODE_ENV=production BUILD_ENV=prod node server/server.js",
+ "start-pre": "cross-env NODE_ENV=production PRE_ENV=true BUILD_ENV=prod node server/server.js",
+ "test": "cross-env NODE_ENV=production BUILD_ENV=test node server/server.js",

- "build:client": "cross-env NODE_ENV=production webpack --config build/webpack.client.config.js --progress --hide-modules",

# mini-css-extract-plugin

webpack4 中不再使用 extract-text-webpack-plugin 而是使用 mini-css-extract-plugin 代替 ，mini-css-extract-plugin 会将 CSS 提取到单独的文件中，为每个包含 CSS 的 JS 文件创建一个 CSS 文件，并且支持 CSS 和 SourceMaps 的按需加载。 与 extract-text-webpack-plugin 相比：

  • 异步加载
  • 没有重复的编译（性能）
  • 更容易使用
  • 特别针对 CSS 开发
- "extract-text-webpack-plugin": "^3.0.2"
+ "mini-css-extract-plugin": "^1.5.0"
1
2

使用方法

// webpack.config.js

const MiniCssExtractPlugin = require('mini-css-extract-plugin');
module.exports = {
  plugins: [new MiniCssExtractPlugin()],
  module: {
    rules: [
      {
        test: /\.css$/i,
        use: [MiniCssExtractPlugin.loader, 'css-loader'],
      }
    ]
  }
};

# CSS抽离

将所有的css抽离到一个文件中，以便使用cdn缓存css文件，这里使用 splitChunks 提供的 cacheGroups 字段

const MiniCssExtractPlugin = require('mini-css-extract-plugin');

module.exports = {
  optimization: {
    splitChunks: {
      cacheGroups: {
        styles: {
          name: 'styles',
          type: 'css/mini-extract',
          // For webpack@4
          // test: /\.css$/,
          chunks: 'all',
          enforce: true,
        }
      }
    }
  },
  plugins: [
    new MiniCssExtractPlugin({
      filename: '[name].css',
    })
  ],
  module: {
    rules: [
      {
        test: /\.css$/,
        use: [MiniCssExtractPlugin.loader, 'css-loader'],
      }
    ]
  }
}

# terser-webpack-plugin

在 webpack4 以上使用 terser (opens new window)来压缩 JavaScript，代替掉 UglifyJsPlugin ，需要注意的是 webpack4 版本需要制定 terser-webpack-plugin 的 v4 版本。

- new webpack.optimize.UglifyJsPlugin({
-     compress: {
-         unused: false,
-         warnings: false
-     },
-     sourceMap: false
- })

+ optimization: {
+     minimize: !isDev,
+     minimizer: [
+         new TerserPlugin({
+             parallel: true,
+             exclude: /\.min\.js$/, // 过滤掉以".min.js"结尾的文件，我们认为这个后缀本身就是已经压缩好的代码，没必要进行二次压缩
+             extractComments: false, // 移除注释
+             minify: (file, sourceMap) => {
+                 const uglifyJsOptions = {
+                     compress: false,
+                     mangle: true,
+                     sourceMap: false,
+                     output: {
+                         comments: false,
+                     }
+                 }
+                 return require('uglify-js').minify(file, uglifyJsOptions)
+             },
+         }),
+     ]
+ }

# cache-loader

cache-loader 允许缓存 loaders 到（默认）磁盘或数据库。

⚠️ 请注意，保存和读取这些缓存文件会有一些时间开销，所以请只对性能开销较大的 loader 使用此 loader。

将 cache-loader 放在耗时较多的 loader 前面即可:

// webpack.config.js
module.exports = {
  module: {
    rules: [
      {
        test: /\.js$/,
        use: ['cache-loader', 'babel-loader']
      }
    ]
  }
};

# thread-loader

npm install --save-dev thread-loader
1

thread-loader 可以将 loader 放在一个独立的 worker 池中运行，使用时，需将此 loader 放置在其他 loader 之前。放置在此 loader 之后的 loader 在 worker 池中运行的 loader 是受到限制的。例如：

  • 这些 loader 不能生成新的文件。
  • 这些 loader 不能使用自定义的 loader API（也就是说，不能通过插件来自定义）。
  • 这些 loader 无法获取 webpack 的配置。

每个 worker 都是一个独立的 node.js 进程，其开销大约为 600ms 左右。同时会限制跨进程的数据交换。

_⚠️ _请仅在耗时的操作中使用此 loader！

示例：

// webpack.config.js
const os = require('os')

const jsWorkerPool = {
    workers: os.cpus().length // 拿到系统CPU的最大核数，让 thread-loader将编译工作灌满所有CPU核
}
// 通过预警 worker 池来防止启动 worker 时的高延时,这会启动池内最大数量的 worker 并把指定的模块加载到 node.js 的模块缓存中。

threadLoader.warmup(jsWorkerPool, ['babel-loader', 'vue-loader']) 

module.exports = {
  module: {
    rules: [{
     test: /\.vue$/,
     use: [{
             loader: 'thread-loader',
             options: jsWorkerPool
           },
           'cache-loader',
           {
             loader: 'vue-loader'
           }
       ]
 			}
    ]
  }
};

# loader.include

         test: /\.vue$/,
